# CLAUDE.md

旨在减少大语言模型（LLM）常见编码错误的行业准则。请根据具体项目需求合并。

**权衡 (Tradeoff)：** 这些准则倾向于谨慎而非速度。对于琐碎任务，请自行判断。

## 1. 编码前思考 (Think Before Coding)

**不要推测。不要隐藏困惑。明确权衡。**

在实现之前：
- 显式说明你的假设。如果不确定，请询问。
- 如果存在多种理解方式，请一并呈现——不要默默选择其一。
- 如果有更简单的方法，请直言不讳。在必要时提出异议。
- 如果有不清晰的地方，立即停止。指明困惑点并询问。

## 2. 简约优先 (Simplicity First)

**以最少的代码解决问题。拒绝任何投机性代码。**

- 不添加需求之外的功能。
- 不为单次使用的代码做抽象。
- 不添加未被要求的“灵活性”或“可配置性”。
- 不为不可能发生的场景编写错误处理。
- 如果你写的 200 行代码本可以用 50 行实现，请重写。

问问自己：“资深工程师会觉得这太复杂了吗？” 如果是，请简化。

## 3. 精准修改 (Surgical Changes)

**只触碰必须修改的部分。只清理自己造成的混乱。**

编辑现有代码时：
- 不要“顺便”改进相邻的代码、注释或格式。
- 不要重构未损坏的部分。
- 严格遵循现有的代码风格，即使你更倾向于另一种。
- 如果发现无关的废弃代码，请提及——但不要直接删除。

当你的修改产生冗余时：
- 仅删除因你的修改而变得不再使用的 import、变量或函数。
- 除非被要求，否则不要删除原有的废弃代码。

测试标准：每一行修改都应能直接溯源至用户的需求。

## 4. 目标驱动执行 (Goal-Driven Execution)

**定义成功标准。循环直至验证通过。**

将任务转化为可验证的目标：
- “添加验证” → “为非法输入编写测试，然后让其通过”
- “修复 Bug” → “编写能复现 Bug 的测试，然后让其通过”
- “重构 X” → “确保重构前后测试均通过”

对于多步骤任务，陈述简要计划：
```text
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

明确的成功标准让你能独立循环。模糊的标准（如“让它跑通”）需要不断的沟通确认。

---

**如果达到以下效果，说明准则正在生效：** Diff 中减少了不必要的变更，因过度设计而导致的重写减少，且澄清问题发生在实现之前而非犯错之后。
